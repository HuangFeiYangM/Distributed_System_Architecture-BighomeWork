## 1. 部署目标与拓扑

### 1.1 目标

- 把 5 个后端微服务（gateway / user / menu / order / pickup）和前端（smart-canteen-web）部署到 K3S 集群；
- 复用宿主机已运行的 **Nacos / MySQL / Redis**，不在集群内重建有状态依赖；
- 提供"离线打包脚本 + 一键部署脚本"，目标电脑可以脱离源码仓库直接部署。

### 1.3 关键约束

- **Pod 内不能用 `127.0.0.1` 访问宿主机**：所有访问宿主机的环境变量统一改为 `host.k3d.internal`（k3d 在节点内自动解析为宿主机 IP）。
- **镜像拉取不走公网**：使用 `imagePullPolicy: IfNotPresent` + `k3d image import` 把本地镜像推入集群内置 registry。
- **命名空间隔离**：所有资源放在独立 namespace `canteen-offline`，不污染默认 `canteen` 命名空间。

## 3. 镜像构建与离线打包

整个流程由 [k3s-version/scripts/prepare-offline-bundle.ps1](../../k3s-version/scripts/prepare-offline-bundle.ps1) 一键完成。脚本职责：

1. `mvn -DskipTests clean package` 在 `smart-canteen/` 下打包 5 个 JAR；
2. 把 5 个可运行 JAR 复制到 `k3s-version/artifacts/backend/`；
3. `npm install && npm run build` 在 `smart-canteen-web/` 下产出 `dist/`，复制到 `k3s-version/artifacts/web/dist`；
4. 用 `k3s-version/dockerfiles/*.artifact.Dockerfile` 构建 6 个镜像（默认 tag `v1`）；
5. `docker save` 导出为 `k3s-version/images/*.tar` 离线包。

### 3.1 一键执行

在仓库根目录：

```powershell
powershell -ExecutionPolicy Bypass -File .\k3s-version\scripts\prepare-offline-bundle.ps1 -ImageTag v1
```

完成后 `k3s-version/` 下生成：

```text
k3s-version/
├── artifacts/
│   ├── backend/   # *.jar
│   └── web/dist/  # 前端静态产物
└── images/
    ├── canteen-gateway-v1.tar
    ├── canteen-user-service-v1.tar
    ├── canteen-menu-service-v1.tar
    ├── canteen-order-service-v1.tar
    ├── canteen-pickup-service-v1.tar
    └── smart-canteen-web-v1.tar
```

把整个 `k3s-version/` 文件夹复制到目标电脑后，无需源码即可部署。

### 3.2 后端镜像 Dockerfile 模板

以网关为例（[k3s-version/dockerfiles/gateway.artifact.Dockerfile](../../k3s-version/dockerfiles/gateway.artifact.Dockerfile)）：

```dockerfile
FROM eclipse-temurin:17-jre
WORKDIR /app
COPY artifacts/backend/canteen-gateway.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java","-jar","/app/app.jar"]
```

5 个后端服务结构相同，仅 JAR 名与 `EXPOSE` 端口不同（user 8081 / menu 8082 / order 8083 / pickup 8084）。

### 3.3 前端镜像 Dockerfile

[k3s-version/dockerfiles/web.artifact.Dockerfile](../../k3s-version/dockerfiles/web.artifact.Dockerfile)：

```dockerfile
FROM nginx:1.27-alpine
COPY artifacts/web/dist/ /usr/share/nginx/html/
COPY nginx/default.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
```

`nginx/default.conf`（[k3s-version/nginx/default.conf](../../k3s-version/nginx/default.conf)）启用 K8s DNS 解析，把 `/api/*` 与 `/ws/*` 反代到集群内的 Gateway Service：

```nginx
resolver kube-dns.kube-system.svc.cluster.local valid=10s ipv6=off;

location /api/ {
    set $gateway_upstream canteen-gateway.canteen-offline.svc.cluster.local:8080;
    proxy_pass http://$gateway_upstream;
    ...
}

location /ws/ {
    set $gateway_upstream canteen-gateway.canteen-offline.svc.cluster.local:8080;
    proxy_pass http://$gateway_upstream;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    ...
}
```

## 4. K8s 资源清单解读

所有 YAML 在 [k3s-version/k8s/](../../k3s-version/k8s/)。

### 4.1 Namespace

[namespace.yaml](../../k3s-version/k8s/namespace.yaml)：

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: canteen-offline
```

### 4.2 后端服务模板（以 user 为例）

[user.yaml](../../k3s-version/k8s/user.yaml) 同时声明 Deployment + Service：

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: canteen-user-service
  namespace: canteen-offline
spec:
  replicas: 1
  selector:
    matchLabels:
      app: canteen-user-service
  template:
    metadata:
      labels:
        app: canteen-user-service
    spec:
      containers:
        - name: canteen-user-service
          image: canteen-user-service:v1
          imagePullPolicy: IfNotPresent
          ports:
            - containerPort: 8081
          env:
            - name: SPRING_CLOUD_NACOS_DISCOVERY_SERVER_ADDR
              value: host.k3d.internal:8848
            - name: SPRING_CLOUD_NACOS_CONFIG_SERVER_ADDR
              value: host.k3d.internal:8848
            - name: SPRING_DATASOURCE_URL
              value: jdbc:mysql://host.k3d.internal:3320/canteen_user?useUnicode=true&characterEncoding=utf-8&serverTimezone=Asia/Shanghai
            - name: SPRING_DATASOURCE_USERNAME
              value: root
            - name: SPRING_DATASOURCE_PASSWORD
              value: root
            - name: SPRING_REDIS_HOST
              value: host.k3d.internal
            - name: SPRING_REDIS_PORT
              value: "6379"
---
apiVersion: v1
kind: Service
metadata:
  name: canteen-user-service
  namespace: canteen-offline
spec:
  selector:
    app: canteen-user-service
  ports:
    - port: 8081
      targetPort: 8081
```

要点：

- **标签与选择器**：Deployment `template.metadata.labels.app` 与 Service `spec.selector.app` 必须一致，集群依赖该标签做端点匹配；
- **环境变量覆盖**：通过 Spring Boot 的 relaxed binding，`SPRING_DATASOURCE_URL` 等于覆盖 `application.yml` 中 `spring.datasource.url`，达到"不改代码、改部署清单"；
- **端口规则**：containerPort = Spring Boot `server.port`；Service `port` / `targetPort` 与之保持一致；
- **不需要 ServiceAccount / RBAC / ConfigMap**：所有可变配置均通过 Nacos 与环境变量解决，YAML 干净。

### 4.3 menu / order / pickup 服务

结构与 user 完全一致，差异仅在：

| 服务 | image tag | containerPort | Service port | 数据库 URL 后段 |
|------|-----------|---------------|--------------|-----------------|
| canteen-menu-service | `canteen-menu-service:v1` | 8082 | 8082 | `canteen_menu` |
| canteen-order-service | `canteen-order-service:v1` | 8083 | 8083 | `canteen_order` |
| canteen-pickup-service | `canteen-pickup-service:v1` | 8084 | 8084 | `canteen_pickup` |

pickup 服务 Service 同样为 `ClusterIP`，WebSocket 通过 Gateway 的 `lb:ws://canteen-pickup-service` 路由进入。

### 4.4 Gateway —— 对外 NodePort

[gateway.yaml](../../k3s-version/k8s/gateway.yaml)：

```yaml
apiVersion: v1
kind: Service
metadata:
  name: canteen-gateway
  namespace: canteen-offline
spec:
  type: NodePort
  selector:
    app: canteen-gateway
  ports:
    - port: 8080
      targetPort: 8080
      nodePort: 30080
```

环境变量除了 Nacos / Redis 外，没有 datasource —— 网关本身不直连 MySQL。

### 4.5 前端 Web

[web.yaml](../../k3s-version/k8s/web.yaml)：

```yaml
apiVersion: v1
kind: Service
metadata:
  name: smart-canteen-web
spec:
  type: NodePort
  selector:
    app: smart-canteen-web
  ports:
    - port: 80
      targetPort: 80
      nodePort: 30173
```

容器内 nginx 监听 80，对外通过 NodePort 30173 暴露。前端的 `/api/*` 与 `/ws/*` 在集群内由 nginx 反代到 Gateway Service。

## 5. 一键部署

### 5.1 推荐方式 —— 部署脚本

在仓库根目录执行 [k3s-version/scripts/deploy-offline.ps1](../../k3s-version/scripts/deploy-offline.ps1)：

```powershell
powershell -ExecutionPolicy Bypass -File .\k3s-version\scripts\deploy-offline.ps1 -ClusterName mycluster -ImageTag v1
```

脚本顺序：

1. `docker load` 载入 6 个离线镜像；
2. `k3d image import` 推入 `mycluster` 集群；
3. 按依赖顺序 `kubectl apply` 7 个 YAML（namespace → user → menu → order → pickup → gateway → web）；
4. 若 `ImageTag` 不是 `v1`，自动 `kubectl set image` 切换镜像版本。

### 5.2 手动逐条执行（与脚本等价）


## 6. 验收清单

### 6.1 集群侧检查

```powershell
kubectl get pods -n canteen-offline -o wide
kubectl get svc  -n canteen-offline
```

期望：

- 6 个 Deployment 都至少有 1 个 Pod 处于 `Running` 状态；
- `READY` 列均为 `1/1`；
- `RESTARTS` 不持续上升（首次冷启动可能有 1~2 次 readiness 重试，属正常）。

### 6.2 Nacos 注册检查

打开 http://127.0.0.1:8848/nacos → 服务管理 → 服务列表，应能看到 5 个服务全部健康注册：

- `canteen-gateway`
- `canteen-user-service`
- `canteen-menu-service`
- `canteen-order-service`
- `canteen-pickup-service`

### 6.3 业务验证

以 NodePort 30080 直接打 Gateway，或先做 port-forward。

```powershell
# 注册一个测试用户
curl -Method POST -Uri http://localhost:30080/api/user/register `
  -Body '{"phone":"13800138000","password":"123456","nickname":"测试","studentNo":"K3S001"}' `
  -ContentType 'application/json'

# 登录
curl -Method POST -Uri http://localhost:30080/api/user/login `
  -Body '{"phone":"13800138000","password":"123456"}' `
  -ContentType 'application/json'
```

期望返回 `{"code":200,"msg":"操作成功","data":{...accessToken...}}`。

### 6.4 前端访问

浏览器打开：

- 前端：`http://localhost:30173`
- 网关：`http://localhost:30080`

正常情况：

- 登录页能加载；
- 登录后可拉到今日菜单（前提是已经在 Nacos 中导入配置 + 在 MySQL 中插入了菜品/菜单/窗口数据）；
- 大屏页面 `/display` 可以建立 WebSocket 连接，商家叫号后立即收到推送。

## 7. k3d 下访问的兜底方式 —— port-forward

k3d 的 NodePort 不一定会自动映射到宿主机 `localhost`。当出现"Pod Running，但浏览器打不开 30080/30173"时，使用 port-forward：

```powershell
# 一个终端：前端
kubectl port-forward -n canteen-offline svc/smart-canteen-web 30173:80

# 另一个终端：网关
kubectl port-forward -n canteen-offline svc/canteen-gateway 30080:8080
```

终端窗口须保持运行，关闭即停止转发。

## 8. 故障排查

### 8.1 Pod CrashLoopBackOff

```powershell
kubectl describe pod -n canteen-offline <pod-name>
kubectl logs -n canteen-offline <pod-name> --tail=200
```

常见原因：

| 现象 | 根因 | 修复 |
|------|------|------|
| 启动报 `Failed to connect to Nacos` | Pod 内地址仍是 `127.0.0.1` | 检查 deployment 中的 `SPRING_CLOUD_NACOS_*` 是否为 `host.k3d.internal:8848` |
| 启动报 `Communications link failure` | MySQL 不通 | `kubectl exec` 进 Pod 后 `nc -zv host.k3d.internal 3320` 测试 |
| 启动报 `Unknown database canteen_xxx` | 库未建 | 重新执行 [database_init/init.sql](../../database_init/init.sql) |
| `ImagePullBackOff` | 镜像未导入集群 | 重跑 `k3d image import canteen-xxx:v1 -c mycluster` |

### 8.2 服务注册不上 Nacos

1. 确认 Nacos 启动完毕（容器日志出现 `Nacos started successfully`）；
2. 在 Pod 内 `nc -zv host.k3d.internal 8848` 与 `9848`（gRPC）；
3. Windows 防火墙是否阻断 8848/9848；
4. Nacos 里的 DataId 是否与 `${spring.application.name}` 完全匹配（区分大小写）。

### 8.3 网关 503 / Bad Gateway

下游服务尚未在 Nacos 出现，或下游 Pod 没起来。先 `kubectl get svc -n canteen-offline` 确认 `canteen-xxx-service` ClusterIP 已分配，再看 Nacos 列表。

### 8.4 WebSocket 不通

- 浏览器控制台报 `WebSocket connection failed`：检查 Gateway 路由表是否包含 `id: pickup-websocket`，以及 nginx `location /ws/` 是否正确反代且带 `Upgrade` 头；
- 大屏不收消息：在 pickup-service Pod 中查看叫号日志确认 `broadcast` 已被调用；当前实现是单实例广播，副本数请保持 `replicas: 1`。

### 8.5 镜像 tag 升级

当镜像由 `v1` 升级到 `v2` 时：

```powershell
docker load -i .\k3s-version\images\canteen-gateway-v2.tar
k3d image import canteen-gateway:v2 -c mycluster
kubectl set image deployment/canteen-gateway canteen-gateway=canteen-gateway:v2 -n canteen-offline
```

`deploy-offline.ps1` 已经把上述步骤集成。
