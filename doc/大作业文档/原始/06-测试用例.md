## 1. 测试范围与方法

### 1.1 范围

本文档覆盖系统对外的 HTTP REST 接口，按业务域划分为 6 大类用例（U/M/O/P/G/S）。所有用例均通过 Gateway `http://127.0.0.1:8080` 触发，Python 集成测试脚本作为执行载体。

### 1.2 方法

| 方法类型 | 工具 | 触发方式 | 断言方式 |
|---------|------|---------|---------|
| 接口集成测试 | Python `urllib` + 自研 `request_json`（缺口脚本同左；WS 脚本另需 `pip install websockets`） | `python tests/run_backend_api_xxx.py` | 校验 HTTP 状态 + 业务体 `code` + 关键字段存在性 |
| 端到端流程测试 | 同上（[run_backend_api_tests.py](../../tests/run_backend_api_tests.py)） | 串联用户登录 → 商家发布菜单 → 用户下单 → 商家流转 → 商家叫号 → 用户核销 | 链路任一环节失败即提前 STOP |

### 1.3 测试账号

| 角色 | 手机号 | 密码 | 数据库角色字段 |
|------|-------|------|----------------|
| 管理员 ADMIN | 13800000000 | admin123 | `sys_user.role = 2` |
| 商家 MERCHANT | 13800000001 | 123456 | `sys_user.role = 1` |
| 普通用户 USER | 13800000002 | user123 | `sys_user.role = 0` |

> 角色字段需在数据库手动设置（注册接口默认 role=0，详见 [tests/后端接口测试手册.md](../../tests/后端接口测试手册.md) §2）。  
> 管理员密码若与上表 `admin123` 不一致，运行 `run_backend_api_gap_tests.py` / `run_backend_api_ws_pickup_test.py` / `run_backend_api_order_timeout_test.py` 时可传 `--admin-password`（或环境变量 `CANTEEN_ADMIN_PASSWORD`），与仓库根目录 [password.md](../../password.md) 对齐即可。

### 1.4 用例编号规则

| 前缀 | 含义 |
|------|------|
| TC-U | 用户域 |
| TC-M | 菜品 / 菜单域 |
| TC-O | 订单域 |
| TC-P | 取餐 / 窗口域 |
| TC-G | 网关 / 边界 / 异常 |
| TC-S | 库存补充 |

### 1.5 通用格式约定

每条用例包含：编号、名称、对应接口、前置条件、输入、预期结果（业务码）、对应脚本断言名。"对应脚本断言名"列指脚本中 `record(...)` 的第一个字符串参数，便于脚本输出与本文档对齐。

## 2. 用户域用例（TC-U）

### TC-U01 用户注册（已存在）

| 项 | 内容 |
|---|---|
| 接口 | `POST /api/user/register` |
| 前置 | 手机号未被占用 |
| 输入 | `{"phone":"13800000002","password":"user123","nickname":"测试学生","studentNo":"2024001001"}` |
| 预期 | `code:200, msg:"操作成功"` |
| 异常分支 | 手机号已存在 → `code:1003`；学工号已存在 → `code:1003` |
| 对应脚本 | `run_backend_api_tests.py` 中 `register(account)` |

### TC-U02 用户登录

| 项 | 内容 |
|---|---|
| 接口 | `POST /api/user/login` |
| 前置 | 用户已注册 |

## 8. 端到端流程用例（happy path 链路）

`run_backend_api_tests.py` 串联以下步骤构成一条完整端到端用例（任一步失败 STOP）：

```mermaid
flowchart LR
    A[USER 登录] --> B[USER /me]
    B --> C[USER /refresh]
    C --> D[MERCHANT 登录]
    D --> E[ADMIN 登录]
    E --> F[MERCHANT 创建菜品 x2]
    F --> G[MERCHANT 发布今日菜单 含两道菜]
    G --> H[USER 查询 /menu/today]
    H --> I[USER 查 menuDishId 详情]
    I --> J[ADMIN 创建窗口]
    J --> K[USER 下单 quantity=2]
    K --> L[USER 查订单详情]
    L --> M["MERCHANT 三连流转 accept→cook→ready"]
    M --> N["PICKUP 查 queue / display"]
    N --> O[MERCHANT 叫号 call]
    O --> P[MERCHANT verify pickupCode]
```

每步骤的脚本输出 `[STEP]` + URL + HTTP 响应体，用于人工核对。
