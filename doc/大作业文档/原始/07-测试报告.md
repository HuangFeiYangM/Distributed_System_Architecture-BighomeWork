## 1. 测试基本信息

### 1.1 测试目的

验证系统在功能、接口契约、状态机、边界处理、异常处理上是否符合 [03-需求规格说明书.md](03-需求规格说明书.md) 的约定，以及在 K3S 部署条件下是否可正常工作。

### 1.2 测试范围

- 5 个微服务对外的全部 REST 接口（用户 / 菜品 / 菜单 / 库存 / 订单 / 统计 / 取餐 / 窗口 / 网关）；
- 端到端业务流程（注册→登录→浏览→下单→流转→叫号→核销）；
- 关键异常分支（缺 Token、越权、空队列、库存冲突、状态机非法）；
- 部分横切能力（路由白名单、JWT 校验、统一响应体、业务码体系）。

### 1.3 测试不包含

- Java 单元测试（`src/test/java` 目录当前仅有 `.gitkeep`，未编写 JUnit 用例）；
- 真正的 WebSocket 客户端收消息断言（脚本未实现 WS 客户端，仅依赖 broadcast 调用是否成功）；
- 订单超时定时取消的实跑等待（耗时 ≥30s，需要专用脚本，目前未自动化）；
- Token 过期 / 黑名单的实际命中（脚本只验证签名错误时的 401）；
- Gateway 限流压力测试（限流配置已就位，但实际 filter 未注册）；
- 大并发性能压测（无 JMeter / k6 性能脚本）。

## 4. 未覆盖项与遗留风险（保留章节）

按需求 06 章 §1.3 所述，以下能力当前 **未被自动化测试覆盖**，按用户要求显式留空但保留在文档中，便于后续补齐：

### 4.1 单元测试

- 状态：5 个微服务下 `src/test/java/` 目录均仅含 `.gitkeep`，未编写任何 JUnit / Mockito 单元测试。
- 影响：业务逻辑的独立验证依赖集成测试脚本；细粒度算法（如 `OrdersAppService.create` 在异常分支的 `restoreStocks` 行为）未被单测固定。
- 建议补齐：`UserServiceTest`、`OrdersAppServiceTest`、`PickupQueueServiceTest`（重点覆盖 enqueue / call / verify 的业务码 4002/4003）、`DailyMenuServiceTest`（重点覆盖 `updateStock` 三种 op 的 5 类异常分支）。

### 4.2 WebSocket 客户端断言

- 状态：脚本以"叫号 / 核销返回 200"作为代理验证；未编写 WS 客户端连 `ws://localhost:8080/ws/pickup` 等待 `type=CALL/PICKED` 消息。
- 影响：当 WebSocket 推送异常（如序列化错误、广播线程死锁）时无法被脚本捕获。
- 建议补齐：使用 Python `websockets` 库或 Node.js 写一个连接客户端，在调用 `/api/pickup/{wid}/call` 之前订阅，断言 1 秒内收到对应 `pickupNo` 的 CALL 消息。

### 4.3 订单超时自动取消（OrderTimeoutScheduler）

- 状态：`@Scheduled(fixedDelay = 30s)`，需要专门的等待型测试。
- 影响：超时取消逻辑（`cancelType=2` + 库存恢复）未被自动化验证。
- 建议补齐：构造 `payDeadline = now - 1s` 的订单（绕过 service.create 的固定 +15min，需额外接口或直接 SQL 注入），等 30~60 秒后断言订单状态 = 5、`menu_dish.stock` 已恢复、`order.cancelType = 2`。

### 4.4 Token 过期与刷新边界

- 状态：脚本只在签名错误时验证 401；未验证 Token 真正过期（24h 后）的行为，也未验证 7 天宽限期外刷新返回 `1005`。
- 建议补齐：临时把 `jwt.expiration` 改成几秒后再跑刷新脚本，或注入伪造的过期 Token。

### 4.5 Gateway 限流

- 状态：`RateLimitProperties` 已读入但未注册具体的限流过滤器实现。
- 影响：高 QPS 下没有限流保护，不能被打压测试覆盖。
- 建议补齐：实现基于 Redis Token Bucket 的 `IpRateLimitFilter / UserRateLimitFilter`，再用 `wrk` 或 `k6` 验证。

### 4.6 性能 / 并发指标

- 状态：未做并发下单的库存超卖测试，未做接口 P95 延迟测量。
- 建议补齐：`wrk -t8 -c64 -d30s` 命中下单接口；用同一 menuDishId 高并发下单观察 `menu_dish.stock` 是否能保持非负、订单数与库存差值是否一致。

### 4.7 跨服务调用降级

- 状态：当前未配置 Feign + Sentinel / Resilience4j 熔断；下游不可用时上游会抛出原始异常。
- 建议补齐：注入 Mock Feign 客户端模拟超时 / 5xx，验证全局异常处理器是否返回有意义的业务码。

## 5. 测试结论

### 5.1 是否通过

**通过**。理由：

- 端到端 happy path（注册→登录→浏览→下单→流转→叫号→核销）全部步骤自动化通过；
- 所有自动化覆盖到的业务接口（用户 / 菜品 / 菜单 / 库存 / 订单 / 取餐 / 窗口 / 统计 / 网关边界）返回值与业务码均符合 [03-需求规格说明书.md](03-需求规格说明书.md) 中的契约；
- 关键异常分支（缺 Token / 无效 Token / 越权 / 资源不存在 / 空队列叫号 / 库存冲突）均按预期返回 4xx 或对应业务码（401/403/2002/2004/3001/4002/4003）；
- 0 个 Blocker / Major / Minor 缺陷；
- 部署侧已通过 K3S 离线部署脚本验证 Pod Running、Nacos 注册成功、网关与前端 NodePort 可访问。

### 5.2 风险评估

| 风险 | 等级 | 说明 |
|------|------|------|
| 单元测试缺失 | 中 | 课程评分细则中"测试覆盖（10 分）"包含单元测试与集成测试两类；当前以集成测试覆盖了主要功能点，但单测维度的得分可能偏低 |
| WebSocket 实测缺失 | 低 | 大屏功能依赖 broadcast，间接由"叫号接口返回 200"间接验证；前端联调阶段已能直观看到消息推送 |
| 订单超时取消未自动化 | 低 | 业务逻辑代码已实现并经过 review；只需追加一个 60 秒等待型脚本即可 |
| 限流缺失 | 低 | 评分要求未强约束限流必须落地，配置已就位为后续接入预留 |

### 5.3 下一步建议

1. 在 `src/test/java/` 目录下补齐每个服务至少 1 套核心 service 类的 JUnit + Mockito 单测；
2. 增加 WS 客户端断言脚本，把 TC-P05a 转入自动化覆盖；
3. 增加 `run_timeout_scheduler_test.py` 验证 TC-O05；
4. 实现 Gateway 限流过滤器后，补充压测脚本并重新出具一份"v1.1"测试报告。
