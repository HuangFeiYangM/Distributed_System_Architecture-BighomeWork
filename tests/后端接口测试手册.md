## 3. 测试范围与顺序

### 3.1 测试模块划分

| 模块 | 测试内容 | 优先级 |
|------|---------|--------|
| 用户服务 | 注册、登录、Token刷新、信息查询/修改 | P0 |
| 菜品服务 | 菜品CRUD、菜单发布、今日菜单查询、库存扣减/恢复 | P0 |
| 订单服务 | 下单、订单查询、状态流转、超时取消、取餐回调 | P0 |
| 取餐服务 | 入队、叫号、核销、队列查询、大屏数据、WebSocket | P0 |
| 网关 | JWT校验、路由转发、白名单、WebSocket代理 | P1 |

### 3.2 推荐测试顺序（存在依赖关系）

```
Step 1: 用户服务（注册 → 登录获取Token）
    ↓
Step 2: 菜品服务（商家登录 → 创建菜品 → 发布菜单）
    ↓
Step 3: 订单服务（用户登录 → 浏览菜单 → 下单）
    ↓
Step 4: 取餐服务（商家接单 → 制作 → 完成 → 叫号 → 核销）
    ↓
Step 5: 边界与异常测试（超时、库存不足、无权限等）
```

## 4. 用户服务测试用例

### 4.1 TC-U01：用户注册

| 项 | 内容 |
|----|------|
| **接口** | `POST /api/user/register` |
| **描述** | 新用户注册 |
| **请求头** | `Content-Type: application/json` |

**请求体**：
```json
{
  "phone": "13800138000",
  "password": "123456",
  "nickname": "张三",
  "studentNo": "2024001002"
}
```

**预期结果**：
```json
{
  "code": 200,
  "msg": "操作成功",
  "data": "注册成功",
  "timestamp": 1714291200000
}
```

**异常测试**：重复手机号注册 → `code: 1001`（用户已存在）

---

### 4.2 TC-U02：用户登录

| 项 | 内容 |
|----|------|
| **接口** | `POST /api/user/login` |
| **描述** | 使用手机号+密码登录，获取 JWT Token |

**请求体**：
```json
{
  "phone": "13800000002",
  "password": "user123"
}
```

**预期结果**：
```json
{
  "code": 200,
  "msg": "操作成功",
  "data": {
    "accessToken": "eyJhbGciOiJIUzI1NiIs...",
    "tokenType": "Bearer",
    "expiresIn": 86400,
    "userId": 3,
    "role": 0
  },
  "timestamp": 1714291200000
}
```

> **重要**：记录返回的 `accessToken`，后续所有需要鉴权的接口都要在 Header 中携带：
> ```
> Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
> ```

---

### 4.3 TC-U03：查询当前用户

| 项 | 内容 |
|----|------|
| **接口** | `GET /api/user/me` |
| **描述** | 根据 Token 查询当前登录用户信息 |
| **请求头** | `Authorization: Bearer {token}` |

**预期结果**：
```json
{
  "code": 200,
  "msg": "操作成功",
  "data": {
    "id": 3,
    "phone": "13800000002",
    "studentNo": "2024001001",
    "nickname": "测试学生",
    "role": 0,
    "status": 1
  }
}
```

---

### 4.4 TC-U04：Token 刷新

| **描述** | 使用旧 Token 换取新 Token |

## 6. 订单服务测试用例

在测试取餐前，需要先创建窗口。使用管理员账号：

```http
POST http://localhost:8080/api/pickup/window
Authorization: Bearer {管理员token}
Content-Type: application/json

{
  "name": "1号档口-盖浇饭",
  "location": "食堂一楼东侧",
  "merchantId": 2,
  "pickupPrefix": "A"
}
```

记录返回的 `windowId`（假设为 1）。

---

|----|------|
| **接口** | `POST /api/order` |

### 6.5 TC-O05：开始制作

| **请求头** | `Authorization: Bearer {商家token}` |

**预期结果**：订单状态变为 `2(制作中)`。

---

### 6.6 TC-O06：制作完成

| 项 | 内容 |
|----|------|
| **接口** | `PUT /api/order/{orderId}/ready` |
| **描述** | 商家标记制作完成，触发入队 |
| **请求头** | `Authorization: Bearer {商家token}` |

**预期结果**：订单状态变为 `3(待取餐)`，同时 Pickup Service 中该订单进入窗口队列。

---

### 6.7 TC-O07：用户取消订单

| 项 | 内容 |
|----|------|
| **接口** | `PUT /api/order/{orderId}/cancel` |
| **描述** | 用户在商家接单前取消订单 |
| **请求头** | `Authorization: Bearer {用户token}` |

**前置条件**：订单状态必须为 `0(已下单)`。

**预期结果**：订单状态变为 `5(已取消)`，库存恢复。

---

### 6.8 TC-O08：超时自动取消（验证）

**测试方法**：
1. 创建一个新订单
2. 等待 15 分钟（或修改数据库 `pay_deadline` 为过去时间）
3. 观察 `OrderTimeoutScheduler` 日志
4. 查询订单状态应变为 `5(已取消)`

> 加速测试：直接在数据库修改 `pay_deadline`：
> ```sql
> UPDATE orders SET pay_deadline = DATE_SUB(NOW(), INTERVAL 1 MINUTE) WHERE id = 10001;
> ```

## 10. 快速参考：curl 命令

```bash
curl http://localhost:8080/api/menu/today
```

```bash
curl -X POST http://localhost:8080/api/order   -H "Content-Type: application/json"   -H "Authorization: Bearer YOUR_TOKEN"   -d '{"windowId":1,"items":[{"menuDishId":1,"quantity":2}]}'
```
